## 🔴 Критичные проблемы

### 1. Потеря позиций по «пресс для топливных брикетов» — нет 301 со старого slug

**Доказательство:**  
Яндекс.Вебмастер (скриншот «Мониторинг запросов»): URL `/oborudovanie/press-briketirujushhij-pbm-2/` мониторится — 60–70 показов в феврале 2026, в марте клики нулевые, позиции падают. Текущий slug в коде: `press-pbm2-dlya-briketov`. В Nginx редиректа не было.

**Что значит:** Яндекс продолжает ранжировать старый URL, который возвращает 404. Весь накопленный ранжирующий сигнал не передаётся на новую страницу.

**Исправление:** Добавлен 301-редирект в `nginx-kirovbelmash.conf`:
```nginx
location = /oborudovanie/press-briketirujushhij-pbm-2 {
    return 301 /oborudovanie/press-pbm2-dlya-briketov;
}
location = /oborudovanie/press-briketirujushhij-pbm-2/ {
    return 301 /oborudovanie/press-pbm2-dlya-briketov;
}
```

**После деплоя:** В Яндекс.Вебмастере → «Переобход URL» отправить `/oborudovanie/press-briketirujushhij-pbm-2` на переиндексацию.

### 2. `utm_referrer` не в Clean-param — дублирующийся URL в Метрике и потенциально в индексе

**Доказательство:**  
Метрика (скриншот «Страницы входа»): URL `/?utm_referrer=https%3F%2Fya.ru%2F` фигурирует как отдельная страница входа — 147 визитов, 98,64% отказов. Директивы `Clean-param` в robots.txt не было для `utm_referrer`.

**Что значит:** Яндекс.Бот может обходить и индексировать главную страницу с параметром `utm_referrer` как отдельный URL — дубль главной.

**Исправление:** В `scripts/generate-sitemap.ts` (шаблон robots.txt) добавлен `utm_referrer`:
```
Clean-param: utm_referrer&etext&utm_source&...
```
Вступит в силу после следующего `npm run build` (robots.txt перегенерируется).

## 🟠 Важные проблемы

### 3. `lastmod` в sitemap.xml — все URL получали одинаковую дату сборки

**Проблема:**  
`scripts/generate-sitemap.ts` использовал `const now = new Date().toISOString()` для всех URL. Яндекс игнорирует одинаковый `lastmod` у всех страниц и хуже приоритизирует переобход.

**Исправление:**  
Блог-посты теперь получают реальную дату публикации (`post.date`). Остальные страницы — дату сборки (нет данных для более точного значения).

---

### 4. `priceValidUntil: "2026-12-31"` хардкод в JSON-LD страниц линий

**Файлы:**
- `src/app/(site)/linii-briketirovaniya/page.tsx`
- `src/app/(site)/linii-granulirovaniya/page.tsx`

**Исправление:**  
Заменено на `${new Date().getFullYear()}-12-31` — автоматически текущий год. `LineProductJsonLd.tsx` уже использовал динамическое значение.

### 5. `/cities/` → 410, но поиск всё ещё шлёт трафик (64 визита из SEO)

**Ситуация:**  
Nginx возвращает 410 Gone для всех `/cities/*` (правильно — doorway-страницы). Но Яндекс ещё не убрал их из индекса, пользователи попадают на «страница не найдена».

**Действие (ручное):**  
Яндекс.Вебмастер → Инструменты → Удаление URL → добавить все `/cities/*` URL для ускоренного удаления из индекса. URL из CSV для удаления:
- `/cities/norilsk/`
- `/cities/himki/`
- `/cities/nefteyugansk/`
- `/cities/bor/`

### 9. Другие slug-и оборудования с возможными старыми URL

Пресс — не единственная единица оборудования, у которой мог смениться slug при миграции с WordPress. Текущие slug-и в `equipmentData.ts` (30 позиций) используют краткие форматы (например, `granuljator-ogm-1-5`, `baraban-sushilniy-avm-1-5`), тогда как WordPress мог использовать более длинные.

**Действие:**  
1. В Яндекс.Вебмастер → Индексирование → Страницы в поиске — выгрузить все `/oborudovanie/*`  
2. Сравнить со списком slug-ов в `equipmentData.ts`  
3. Для каждого несовпадения добавить 301 в `nginx-kirovbelmash.conf`
