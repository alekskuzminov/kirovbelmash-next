# Журнал архитектурных решений (CRM)

Каждая запись — почему было выбрано то или иное решение. При сомнениях/изменениях курса в будущем — читать тут.

Формат: дата + контекст + решение + последствия.

## 2026-04-27 — CRM строим внутри Next.js, а не отдельный сервис

**Контекст:** Можно было вынести CRM в отдельный backend (NestJS / Fastify) с собственным фронтом. Или оставить всё в монорепо Next.js.

**Решение:** Всё внутри `kirovbelmash-next`. Маршруты `/admin/*`, та же база PostgreSQL, та же авторизация Next-стиль.

**Почему:**
- Нет нужды в отдельной инфраструктуре — один VPS, один процесс PM2.
- Заявки с сайта попадают в БД через тот же `/api/contact` — ноль сетевых хопов.
- Один код, одна сборка, один деплой.
- Если в будущем понадобится разделить — рефакторинг возможен.

**Последствия:** Sidebar/auth компоненты тянутся в один бандл с публичным сайтом. Это допустимо — Next.js делает code-splitting по роутам, на публичных страницах их нет.

## 2026-04-27 — PostgreSQL на том же VPS, не managed

**Контекст:** Варианты — Supabase Free, Neon Free, или Postgres напрямую на VPS.

**Решение:** PostgreSQL установлен на VPS в systemd, без Docker.

**Почему:**
- Полный контроль и приватность данных клиентов.
- Latency около нуля (localhost).
- Нет зависимости от стороннего бесплатного тарифа, который может стать платным.
- При объёме «несколько менеджеров, тысячи сделок» нагрузка на VPS не заметна.

**Последствия:**
- Бэкапы — наша забота. Нужен cron + дамп в S3.
- Мониторинг — наша забота.
- При масштабировании на других клиентов — managed станет проще.

## 2026-04-27 — Prisma 6, не 7

**Контекст:** Prisma 7 уже релизнулась (на момент работы — 7.8.0), npx по умолчанию подтягивает её.

**Решение:** Пин на `^6` в `package.json`, на VPS — `npx prisma@6`.

**Почему:** В Prisma 7 удалена поддержка `url = env("DATABASE_URL")` в `schema.prisma` — теперь требуется отдельный `prisma.config.ts` с экспортом адаптера. Миграция возможна, но это лишняя работа сейчас.

**Последствия:** При установке через `npx prisma <cmd>` (без `@6`) кэш npx может подтянуть 7 и сломать команду. Всегда использовать `npx prisma@6` или `./node_modules/.bin/prisma`.

## 2026-04-27 — NextAuth v4, JWT, Credentials

**Контекст:** NextAuth v5 в beta. Database sessions vs JWT. OAuth-провайдеры vs Credentials.

**Решение:** v4 (стабильная), JWT-стратегия, только Credentials (email + password).

**Почему:**
- v4 — production-ready, у v5 могут быть breaking changes до релиза.
- JWT — не нужна таблица сессий, одна сетевая операция меньше при каждом запросе.
- Credentials — у нас закрытый круг менеджеров, OAuth избыточен. Email/пароль управляются админом.

**Последствия:** Logout не инвалидирует JWT мгновенно (он живёт до истечения). Для нашего сценария это допустимо. При компрометации — менять `NEXTAUTH_SECRET`.

## 2026-04-27 — `proxy.ts` вместо `middleware.ts`

**Контекст:** Next.js 16 переименовал `middleware.ts` в `proxy.ts` с именованным экспортом `proxy`. Сборка ругается deprecation-warning.

**Решение:** Файл `src/proxy.ts` с `export const proxy = withAuth(...)`.

**Почему:** Соответствие Next.js 16. При обратной миграции на 15 — переименовать.

## 2026-04-27 — `--legacy-peer-deps` в `npm install`

**Контекст:** `next-auth@4` хочет `nodemailer@^7`, в проекте — `nodemailer@^8` (для `/api/contact`).

**Решение:** Флаг `--legacy-peer-deps` в локальной установке и в `deploy.sh`.

**Почему:** `next-auth` использует nodemailer только в Email-провайдере, который мы не подключали. v8 совместима по нашим use-case. Понижать nodemailer ради peer-зависимости, которая не используется — нерационально.

**Последствия:** При апгрейде `next-auth` до v5 (или появлении next-auth поддерживающего `nodemailer@8`) — флаг можно убрать.

## 2026-04-27 — Контакт и сделка — отдельные сущности

**Контекст:** В нашем B2B повторные обращения редкие. Можно было хранить контактные данные прямо в Deal.

**Решение:** Отдельная модель `Contact` с relation 1:N на `Deal`.

**Почему:**
- Когда повторное обращение всё-таки случается — менеджер видит всю историю по клиенту.
- Документы клиента (рамочные договоры, реквизиты) логично хранить на контакте, а не на каждой сделке.
- Это стандарт CRM, переносится в любую другую систему без переделки данных.

**Последствия:** При создании сделки из формы сайта — нужна логика «найти контакт по телефону или создать новый».

## 2026-04-28 — PDF-only для предпросмотра документов в карточке сделки

**Контекст:** Нужен предпросмотр КП и спецификаций прямо в карточке сделки без скачивания. Рассматривались Google Docs Viewer, Microsoft Office Online Viewer, LibreOffice Online.

**Решение:** Принимать только PDF. Рендерить через нативный `<iframe>` браузера — без каких-либо внешних сервисов.

**Почему:**
- Google Docs Viewer и Microsoft Office Online — зависимость от внешних сервисов с геополитическими рисками (юрисдикция РФ).
- PDF рендерится браузером нативно — ноль внешних зависимостей, работает офлайн.
- КП клиентам всегда отправляют в PDF — это де-факто стандарт. DOCX редактируют локально, на выходе PDF.
- Файлы хранятся на собственном Beget S3 / CDN (`cdn.kirovbelmash.ru`), URL публичный.

**Последствия:** Менеджеры должны загружать только PDF. DOCX/XLSX не поддерживаются в превью — при необходимости расширить можно через PDF.js (бандлится локально) без внешних сервисов.

## 2026-04-28 — Реструктуризация URL админки: /admin/crm/*

**Контекст:** Изначально CRM жила прямо под `/admin/deals`, `/admin/contacts`. Админка задумывалась как расширяемая площадка для разных разделов (CRM, контент, настройки).

**Решение:** CRM перенесена на `/admin/crm/*`. `/admin/` — хаб-страница с навигацией по разделам. Старые URL отдают редиректы.

**Почему:** Масштабируемость — будущие разделы (контент, пользователи) встанут рядом с CRM без конфликтов. Хаб-страница даёт общую точку входа.

**Последствия:** Все `revalidatePath`, `href` и `router.push` обновлены. Backward-compatible через редиректы.

## 2026-04-28 — Кастомные поля сделки — не строим, добавляем через схему

**Контекст:** Обсуждалась система динамических кастомных полей (`DealCustomField` + `DealCustomValue`) по аналогии с amoCRM.

**Решение:** Не строить инфраструктуру кастомных полей. Новые поля добавляются через `schema.prisma` по мере реальной необходимости.

**Почему:** Один клиент, разработка ведётся через код. Динамические поля — фича для мультитенантного SaaS. Добавить колонку в схему + инпут в форму занимает 15 минут, а инфраструктура кастомных полей — несколько часов без реальной пользы сейчас.

**Последствия:** При появлении нового поля — `schema.prisma` + `prisma generate` + `db push` + UI. Шаблон простой и воспроизводимый.

## 2026-04-28 — WON/LOST через drag-to-zone, не через финальные этапы воронки

**Контекст:** Можно было добавить специальные этапы «Выиграно» / «Проиграно» в конец воронки (как в amoCRM по умолчанию).

**Решение:** Enum `DealResult { WON, LOST }` + поля `closedAs` и `closedAt` на модели `Deal`. Три зоны сброса (удалить / не реализована / успешно завершена) появляются внизу экрана при перетаскивании карточки.

**Почему:**
- Закрытые сделки не засоряют воронку — их убирает фильтр `closedAt: null`.
- В будущем можно строить аналитику WON/LOST отдельно от воронки.
- UX ближе к amoCRM — менеджер привык к этому паттерну.

**Последствия:** Удалённые сделки (`deletedAt`) и закрытые (`closedAt`) — разные состояния. Для аналитики восстанавливать через `closedAs = 'WON'`. Закрытые сделки нигде не отображаются — нужна отдельная страница «Архив» в будущем.

## 2026-04-28 — Название автосделки: message → extra → fallback

**Контекст:** При автосоздании сделки из формы сайта нужно осмысленное название. Вариант «Заявка с сайта — {Имя}» дублирует поля контакта и источника.

**Решение:** `title = trim60(message) || trim60(extra values joined) || 'Входящая заявка'`. Обрезка до 60 символов с «…».

**Почему:** Название должно отражать *что хочет клиент*, а не *кто он* — это уже видно через контакт. `message` и `extra` (параметры калькулятора) содержат реальный запрос.

**Последствия:** Если форма не передаёт ни message, ни extra — сделка называется «Входящая заявка». Менеджер переименует после первого контакта.

## 2026-04-28 — Источники сделок: «Сайт» vs «Вручную» как дефолты

**Контекст:** Нужно различать автоматически созданные сделки и ручные без явного указания источника.

**Решение:** В `/api/contact` — `source = data.source || 'Сайт'`. В `CreateDealModal` — `useState('Вручную')` как дефолт, менеджер может изменить.

**Почему:** Минимальная идентификация источника без обязательного заполнения. Строки, а не enum — менеджер вводит произвольный источник (звонок, выставка, реферал).

**Последствия:** Аналитика по источникам будет строковой, не типизированной. При необходимости — ввести enum позже.

## 2026-04-27 — Документы привязаны и к Deal, и к Contact (раздельно)

**Контекст:** Можно было хранить все документы только на уровне контакта.

**Решение:** `Document` имеет nullable `dealId` и `contactId`. Документ привязывается к одному из двух (или к обоим — гибридно).

**Почему:** КП и счета — это документы конкретной сделки. Рамочный договор, реквизиты — документы клиента. Разделение помогает менеджеру быстрее ориентироваться.

**Последствия:** В UI нужно явно показывать «документы сделки» и «документы клиента» как разные секции.
